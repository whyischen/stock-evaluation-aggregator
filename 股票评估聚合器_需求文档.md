## 1. 项目概述

### 1.1 项目定义

**股票评估聚合器**（Stock Evaluation Aggregator，以下简称 SEA）是一个面向个人投资者和量化研究者的**开源能力编排平台**。其核心设计哲学是：

> **SEA 本身不做评估判断，而是作为"指挥官"，将评估任务分发给若干已有开源项目，收集各方结论，并以统一视图呈现综合评估结果。**

### 1.2 核心价值

- **聚合多源判断**：汇集基本面分析、技术分析、量化因子、AI 研究等不同维度的评估视角
- **降低使用门槛**：用户无需分别学习和配置多个框架，通过统一接口即可驱动多个评估引擎
- **灵活扩展**：通过插件化设计，可随时添加新的评估源或停用不需要的源，无需改动核心代码

### 1.3 目标用户

| 用户类型 | 使用场景 |
|---|---|
| 个人量化研究者 | 对标的进行多角度交叉验证，提升投研效率 |
| AI Agent 开发者 | 将 SEA 作为工具节点嵌入更大的 Investment Agent 工作流 |
| 平台集成方 | 将多个开源框架的能力包装后对外提供 API 服务 |

## 2. 背景与动机

### 2.1 现有开源项目的能力与局限

目前量化投资和 AI 金融领域已有若干高质量开源项目，各自拥有独特的评估方法论，但也各自存在使用成本：

- 每个项目有独立的安装、配置、依赖环境
- 输出格式不统一，难以横向比较
- 单一框架的评估结论存在方法论偏差，缺乏交叉验证

### 2.2 SEA 的定位

SEA 不是对上述项目的重复造轮子，而是在其之上构建一层**适配与编排层**：

```
用户请求
    │
    ▼
┌─────────────────────────────────────┐
│        SEA 编排核心（本项目）         │
│  ┌──────────────┐  ┌─────────────┐  │
│  │  插件注册表   │  │  结果聚合器  │  │
│  └──────────────┘  └─────────────┘  │
└─────────┬───────────────────────────┘
          │ 分发
    ┌─────┴──────┬──────────────┬───────────────┐
    ▼            ▼              ▼                ▼
[TradingAgents] [TradingAgents-CN] [Qlib]   [RD-Agent]
 LLM多智能体     A股增强版          量化ML平台   自动化因子研究
    评估             评估              评估          评估
```

## 3. 开源评估源分析

以下是 SEA 初始集成的 4 个目标开源项目的能力摘要，作为插件设计的依据。

### 3.1 TradingAgents（TauricResearch）

| 属性 | 说明 |
|---|---|
| **技术路线** | LangGraph 多智能体，LLM 驱动 |
| **评估方法** | 基本面分析师 + 情绪分析师 + 技术分析师 → 多空辩论 → 交易员综合 → 风险管理 → 组合经理 |
| **数据来源** | yFinance、Alpha Vantage、StockTwits、Reddit |
| **输出内容** | 买/卖/持有建议、置信度、分析报告（Markdown）、风险评估 |
| **支持市场** | 所有 yFinance 覆盖的市场（含 A 股 .SS/.SZ） |
| **LLM 支持** | OpenAI、Anthropic、Gemini、DeepSeek、Qwen、GLM、Ollama 等 |
| **调用方式** | Python API：`TradingAgentsGraph.propagate(ticker, date)` |
| **特点** | 结果存在 LLM 随机性，同一标的不同运行可能结论不同；适合定性分析 |

### 3.2 TradingAgents-CN（hsliuping）

| 属性 | 说明 |
|---|---|
| **技术路线** | TradingAgents 的中文增强 Fork |
| **评估方法** | 同 TradingAgents，针对 A 股市场优化 |
| **数据来源** | 增加了国内数据源（AKShare、同花顺等） |
| **输出内容** | 中文评估报告、A 股专项分析 |
| **支持市场** | A 股（沪深两市）为主 |
| **特点** | 对国内 LLM（文心、千问等）和数据源支持更完善；是 TradingAgents 的互补选项 |

### 3.3 Microsoft Qlib

| 属性 | 说明 |
|---|---|
| **技术路线** | AI 量化投资平台，机器学习驱动 |
| **评估方法** | Alpha 因子挖掘 → ML 模型训练（LightGBM/LSTM/Transformer 等）→ 回测 → 组合优化 |
| **数据来源** | Qlib 标准数据集（CSI300/500、NASDAQ 等），支持自定义数据 |
| **输出内容** | 预测收益率（Alpha 信号）、回测绩效指标（夏普比率、最大回撤、年化收益）、因子 IC 值 |
| **支持市场** | A 股、美股（需自行获取数据） |
| **调用方式** | Python API，配置化驱动 |
| **特点** | 量化/ML 视角，确定性强，可重复，适合定量分析 |

### 3.4 Microsoft RD-Agent（Q 模式）

| 属性 | 说明 |
|---|---|
| **技术路线** | LLM 驱动的自动化 R&D 框架，Qlib 为后端执行引擎 |
| **评估方法** | 自动读取金融文献 → 假设生成 → 因子代码生成 → Qlib 回测验证 → 迭代优化 |
| **输出内容** | 新发现的 Alpha 因子、因子表现报告、策略建议 |
| **特点** | 相当于"AI 量化研究员"，能自动发现传统框架找不到的 Alpha；每次运行成本约 $10；结果最具探索性 |
| **依赖** | 依赖 Qlib 环境；适合周期性深度研究，而非实时查询 |

### 4.2 核心模块说明

#### EvaluationTask（评估任务）

用户发起的一次评估请求的数据结构，包含：
- 标的代码列表（如 `["600519.SS", "AAPL"]`）
- 评估日期
- 启用的插件列表（`null` 表示使用所有已启用插件）
- 各插件的可选配置覆盖

#### PluginRegistry（插件注册表）

负责插件的生命周期管理：
- 插件的注册（install）、启用（enable）、禁用（disable）、卸载（uninstall）
- 插件元信息的持久化（存储于本地配置文件 `plugins.yaml`）
- 插件可用性检查（依赖环境验证）

#### Dispatcher（调度器）

负责将 EvaluationTask 分发给启用的插件：
- 支持**并发执行**（多插件并行，适合响应较慢的 LLM 类插件）
- 支持**超时控制**（单插件超时不影响整体结果）
- 支持**降级策略**（插件失败时记录错误，不中断整体流程）

#### Aggregator（结果聚合器）

将多个插件的 `PluginResult` 合并为 `EvalReport`：
- 标准化各插件的方向建议（买/卖/持有 → `+1/0/-1`）
- 计算加权综合评分（权重可配置）
- 标识各插件结论的共识度与分歧度
- 不做最终投资决策，仅呈现综合视图

## 5. 功能需求

### FR-01：插件注册与发现

**描述：** 系统提供插件注册机制，支持内置插件（TradingAgents 等）和第三方插件的注册。

**验收标准：**
- 可通过命令行或 API 查看所有已注册插件及其状态
- 注册信息持久化，重启后不丢失
- 提供插件健康检查命令（验证依赖是否满足）

### FR-02：插件启用与禁用

**描述：** 用户可随时启用或禁用某个插件，无需卸载或修改代码。

**验收标准：**
- 禁用的插件不参与评估任务分发
- 启用/禁用操作立即生效
- 支持按标的市场自动路由（如 A 股优先使用 TradingAgents-CN）

**示例操作：**
```bash
sea plugin disable qlib       # 禁用 Qlib 插件
sea plugin enable tradingagents-cn  # 启用 CN 版
sea plugin list               # 查看所有插件状态
```

### FR-03：股票评估任务执行

**描述：** 用户可以对指定标的发起评估，系统自动调用所有启用插件并收集结果。

**验收标准：**
- 支持单标的和多标的批量评估
- 支持指定评估日期（历史评估）或当前日期（实时评估）
- 评估过程提供进度反馈
- 部分插件失败时，整体任务继续执行并标记失败来源

**示例操作：**
```bash
sea evaluate --ticker 600519.SS --date 2026-06-01
sea evaluate --ticker AAPL,MSFT --plugins trading_agents,qlib
```

### FR-04：统一评估报告输出

**描述：** 所有评估结果以统一的 `EvalReport` 格式呈现，支持多种输出格式。

**验收标准：**
- 支持终端表格输出（人工阅读）
- 支持 JSON 输出（程序集成）
- 支持 Markdown 报告生成（文档存档）
- 报告中明确标注各插件的方向建议、置信度、执行时间

### FR-05：评估结果持久化

**描述：** 评估结果可以保存到本地，支持历史查询和对比。

**验收标准：**
- 结果保存为结构化文件（SQLite 或 JSON Lines）
- 支持按标的、日期、插件查询历史记录
- 支持导出为 CSV 供外部分析

### FR-06：插件配置管理

**描述：** 每个插件有独立的配置项（LLM 提供商、API Key、数据源等），可通过配置文件或环境变量管理。

**验收标准：**
- 提供配置文件模板（`config.yaml`）
- 敏感配置（API Key）支持从环境变量读取
- 配置变更无需重启（除依赖配置外）

### FR-07：Python API 调用支持

**描述：** SEA 提供 Python SDK，可被更大的 AI Agent 工作流（如 LangGraph、LangChain）调用。

**验收标准：**
- 提供 `StockEvaluator` 类，支持 `async evaluate(ticker, date)` 方法
- 返回类型为 `EvalReport` Pydantic 对象
- 支持作为 LangChain Tool / LangGraph Node 使用

### FR-08：第三方插件开发支持

**描述：** 开发者可以按照标准接口开发新的评估源插件并注册到 SEA。

**验收标准：**
- 提供插件开发文档和接口规范
- 提供插件开发脚手架（`sea plugin new my_plugin`）
- 提供插件打包和安装机制（`sea plugin install ./my_plugin`）

## 6. 非功能需求

### NFR-01：可扩展性

- 添加新插件时，核心代码（编排层）无需修改，遵循开闭原则
- 插件以 Python 包形式存在，安装即注册

### NFR-02：容错性

- 单插件异常（超时、崩溃、依赖缺失）不影响其他插件的执行
- 所有插件错误均被捕获、记录，并在报告中标注
- LLM 类插件失败时支持重试（可配置重试次数）

### NFR-03：可观测性

- 提供每次评估的执行日志（各插件耗时、状态）
- 支持结构化日志输出（便于 ELK/Grafana 集成）

### NFR-04：安全性

- API Key 等敏感配置不得出现在日志中
- 插件执行时的网络访问受插件本身控制，SEA 不代理数据请求

### NFR-05：性能

- LLM 类插件（TradingAgents 系列）并发执行，总体等待时间 ≤ 最慢单插件时间 + 10%
- Qlib 信号查询（已有预计算结果）延迟 < 5 秒

## 7. 插件接口规范

### 7.1 核心接口定义

每个评估源插件必须实现以下 Python 抽象基类：

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional
from datetime import date
from enum import Enum

class Direction(Enum):
    BUY   = 1    # 看多
    HOLD  = 0    # 持有/中性
    SELL  = -1   # 看空

@dataclass
class PluginResult:
    plugin_id: str              # 插件唯一标识
    ticker: str                 # 标的代码
    eval_date: date             # 评估日期
    direction: Direction        # 方向建议
    confidence: float           # 置信度 [0.0, 1.0]
    summary: str                # 简要结论（≤ 200字）
    detail: Optional[dict]      # 原始详细输出（各插件自定义结构）
    latency_ms: int             # 执行耗时（毫秒）
    error: Optional[str] = None # 执行错误（None 表示成功）

class BaseEvaluatorPlugin(ABC):

    @property
    @abstractmethod
    def plugin_id(self) -> str:
        """插件唯一标识，如 'tradingagents', 'qlib'"""

    @property
    @abstractmethod
    def display_name(self) -> str:
        """展示名称，如 'TradingAgents (TauricResearch)'"""

    @property
    @abstractmethod
    def supported_markets(self) -> list[str]:
        """支持的市场，如 ['US', 'A_SHARE', 'ALL']"""

    @abstractmethod
    def is_available(self) -> bool:
        """检查插件依赖是否满足（用于健康检查）"""

    @abstractmethod
    async def evaluate(self, ticker: str, eval_date: date, **kwargs) -> PluginResult:
        """执行评估，返回标准化结果"""

    def get_default_config(self) -> dict:
        """返回插件默认配置（可选实现）"""
        return {}
```

## 8. 结果聚合规范

### 8.1 EvalReport 结构

```python
@dataclass
class EvalReport:
    ticker: str
    eval_date: date
    generated_at: datetime

    # 各插件原始结果
    plugin_results: list[PluginResult]

    # 聚合评分（仅当 ≥2 个插件成功返回时计算）
    aggregated_score: Optional[float]   # [-1.0, 1.0]，负值看空，正值看多
    consensus_level: Optional[str]      # "strong" / "moderate" / "diverged"
    direction_summary: Optional[str]    # "2买/1持/0卖" 等文字描述

    # 执行统计
    total_plugins: int
    succeeded_plugins: int
    failed_plugins: int
    total_latency_ms: int
```

### 8.2 聚合算法

综合评分 = Σ（插件方向值 × 置信度 × 权重） / Σ（权重）

其中：
- 插件方向值：BUY=+1, HOLD=0, SELL=-1
- 权重在 `config.yaml` 中配置（默认各插件权重相等）
- 失败插件不参与聚合计算

**共识度判定：**

| 条件 | 共识度 |
|---|---|
| 所有成功插件方向一致 | `strong`（强共识） |
| >60% 插件方向一致 | `moderate`（中等共识） |
| 无多数方向 | `diverged`（意见分歧） |

## 9. 技术选型建议

| 模块 | 建议技术 | 理由 |
|---|---|---|
| 编程语言 | Python 3.11+ | 与所有目标开源项目生态一致 |
| 异步执行 | `asyncio` + `asyncio.gather()` | 并发调用多插件 |
| 插件管理 | 基于 `importlib` + `entry_points` | Python 标准插件机制 |
| 配置管理 | `pydantic-settings` + YAML | 类型安全，支持环境变量 |
| 数据持久化 | SQLite（内置）/ JSON Lines | 零依赖，易迁移 |
| CLI 界面 | `Typer` + `Rich` | 现代化终端 UI |
| Python API | `Pydantic v2` 模型 | 强类型，易序列化 |
| 打包发布 | `uv` + `pyproject.toml` | 现代 Python 包管理 |

## 11. 风险与约束

### 11.1 依赖风险

| 风险 | 描述 | 缓解措施 |
|---|---|---|
| 上游项目 API 变更 | TradingAgents 等项目仍在快速迭代，接口可能变化 | 插件层做版本适配；在 `plugin.yaml` 中声明兼容版本范围 |
| Qlib 数据依赖 | Qlib 需要本地预先下载数据集，冷启动成本高 | 提供数据初始化脚本；文档中明确说明 |
| RD-Agent 执行时间 | 单次运行可能耗时数十分钟到数小时 | 设计为异步任务，不阻塞其他插件；超时后跳过 |

### 11.2 设计约束

- **SEA 不持有任何评估逻辑**：所有评估判断由插件（开源项目）完成，SEA 只做转发和展示
- **SEA 不提供交易执行能力**：仅输出评估结论，无任何下单或资金操作
- **免责声明**：SEA 聚合的所有评估结论仅供研究参考，不构成投资建议

### 11.3 开放问题（待决策）

1. **A 股 vs 美股路由策略**：当 TradingAgents 和 TradingAgents-CN 同时启用时，如何分配 A 股评估任务？（建议：A 股优先 CN 版，海外标的使用原版）
2. **Qlib 运行模式**：插件是调用已有预计算的信号（快速），还是实时跑模型（慢但最新）？
3. **结果缓存策略**：同一标的、同一日期的评估是否缓存？LLM 类插件每次结果不同，缓存意义有限
4. **第三方数据源许可**：确认各开源项目使用的数据源（yFinance、Alpha Vantage 等）在商用/部署场景下的许可情况
