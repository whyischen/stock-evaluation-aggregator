## 1. 项目概述

### 1.1 项目定义

**股票评估聚合器**（Stock Evaluation Aggregator，以下简称 SEA）是一个面向个人投资者和量化研究者的**开源能力编排平台**。其核心设计哲学是：

> **SEA 本身不做评估判断，而是作为"指挥官"，将评估任务分发给若干已有开源项目，收集各方结论，并以统一视图呈现综合评估结果。**

### 1.2 核心价值

- **聚合多源判断**：汇集基本面分析、技术分析、量化因子、AI 研究等不同维度的评估视角
- **降低使用门槛**：用户无需分别学习和配置多个框架，通过统一接口即可驱动多个评估引擎
- **灵活扩展**：通过插件化设计，可随时添加新的评估源或停用不需要的源，无需改动核心代码

### 1.3 目标用户

| 用户类型 | 使用场景 |
|---|---|
| 个人量化研究者 | 对标的进行多角度交叉验证，提升投研效率 |
| AI Agent 开发者 | 将 SEA 作为工具节点嵌入更大的 Investment Agent 工作流 |
| 平台集成方 | 将多个开源框架的能力包装后对外提供 API 服务 |

## 2. 背景与动机

### 2.1 现有开源项目的能力与局限

目前量化投资和 AI 金融领域已有若干高质量开源项目，各自拥有独特的评估方法论，但也各自存在使用成本：

- 每个项目有独立的安装、配置、依赖环境
- 输出格式不统一，难以横向比较
- 单一框架的评估结论存在方法论偏差，缺乏交叉验证

### 2.2 SEA 的定位

SEA 不是对上述项目的重复造轮子，而是在其之上构建一层**适配与编排层**。

### 2.3 核心交互模式

SEA 所有评估插件遵循统一的核心交互模式：

```
evaluate(ticker, date) → PluginResult(direction, confidence, summary)
```

- **输入**：股票代码（ticker）+ 评估日期（date）
- **输出**：标准化的评估结果，包含方向建议（BUY/HOLD/SELL）、置信度、摘要

各开源项目对此模式的契合度分析详见第 3 章。

## 3. 开源评估源深度分析

本章对 SEA 计划集成的开源项目，逐一分析其与核心交互模式 `evaluate(ticker, date)` 的契合度。

### 3.1 契合度总览

| 项目 | 契合度 | 输入模型 | 输出模型 | 调用方式 | 冷启动 | 适配复杂度 |
|---|---|---|---|---|---|---|
| **TradingAgents** | ✅ 完全符合 | 直接接受 ticker + date | 买卖建议 + 置信度 | 单次 `propagate()` 调用 | 无需预计算 | 低 |
| **TradingAgents-CN** | ✅ 完全符合 | 直接接受 ticker + date | 买卖建议 + 中文报告 | 同 TradingAgents API | 无需预计算 | 低 |
| **FinRobot** | ✅ 完全符合 | 接受 ticker，多智能体分析 | 投资建议 + 研报 | Agent 流水线调用 | 无需预计算 | 低 |
| **FinGPT-Forecaster** | ✅ 完全符合 | 接受 ticker | 涨/跌方向预测 | 单次 LLM 调用 | 无需预计算 | 低 |
| **Qlib** | ⚠️ 有条件符合 | 接受股票池，非单 ticker | Alpha 信号，需二次转换 | 配置驱动，非直接调用 | 高（需下载数据 + 训练模型） | 中 |
| **RD-Agent** | ❌ 范式不匹配 | 输入是研究任务，非 ticker | 输出是因子/策略，非个股评估 | 长任务（数十分钟~小时） | 极高，依赖 Qlib 环境 | 极高 |

### 3.2 TradingAgents（TauricResearch）— 天然契合

| 属性 | 说明 |
|---|---|
| **技术路线** | LangGraph 多智能体，LLM 驱动 |
| **评估方法** | 基本面分析师 + 情绪分析师 + 技术分析师 → 多空辩论 → 交易员综合 → 风险管理 → 组合经理 |
| **数据来源** | yFinance、Alpha Vantage、StockTwits、Reddit |
| **输出内容** | 买/卖/持有建议、置信度、分析报告（Markdown）、风险评估 |
| **支持市场** | 所有 yFinance 覆盖的市场（含 A 股 .SS/.SZ） |
| **LLM 支持** | OpenAI、Anthropic、Gemini、DeepSeek、Qwen、GLM、Ollama 等 |
| **调用方式** | Python API：`TradingAgentsGraph.propagate(ticker, date)` |
| **核心优势** | 接口与 SEA 目标模式完全一致，适配层仅需做字段解析（从 `state["final_trade_decision"]` 提取结论），几乎无范式阻抗 |
| **注意事项** | LLM 随机性导致同一标的不同运行结论可能不同；单次耗时 1~5 分钟 |

**适配思路**：直接调用 `propagate(ticker, date)`，从返回的 state 中解析 `final_trade_decision` 字段，映射为 `Direction` 枚举和 `confidence` 值。

| **注意事项** | 本质是 Fork，接口一致。两者同时启用时 A 股路由策略需在 SEA 层决策（见 13.1 已决策事项） |

### 3.4 FinRobot（AI4Finance Foundation）— 完全符合（新增候选）

| 属性 | 说明 |
|---|---|
| **技术路线** | AI Agent 金融分析平台，基于 AutoGen 多智能体 |
| **评估方法** | 多层 Chain-of-Thought，结合定量 + 定性分析 |
| **数据来源** | SEC 财报、公司公告、财报电话会议等多数据源 |

| **定位建议** | 偏重基本面深度研报，适合 A/H 股深度分析场景，作为 TradingAgents 系列的补充选项 |

### 3.5 FinGPT-Forecaster（AI4Finance Foundation）— 完全符合（新增候选）

| 属性 | 说明 |
|---|---|
| **技术路线** | 金融领域微调 LLM，轻量级方向预测 |
| **评估方法** | 单次 LLM 调用，输出综合分析 + 下周股价走势预测 |
| **输出内容** | 涨/跌方向 + 综合分析 |
| **GitHub Stars** | 14K+ |
| **核心优势** | 依赖比 TradingAgents 小得多，响应更快 |
| **定位建议** | 轻量级情绪/方向信号，适合需要低延迟的场景；分析深度有限，非完整多角度研报 |

### 3.6 Microsoft Qlib — 有条件符合（需补全生命周期管理）

| 属性 | 说明 |
|---|---|
| **技术路线** | AI 量化投资平台，机器学习驱动 |
| **评估方法** | Alpha 因子挖掘 → ML 模型训练（LightGBM/LSTM/Transformer 等）→ 回测 → 组合优化 |
| **数据来源** | Qlib 标准数据集（CSI300/500、NASDAQ 等），支持自定义数据 |
| **输出内容** | 预测收益率（Alpha 信号）、回测绩效指标（夏普比率、最大回撤、年化收益）、因子 IC 值 |
| **支持市场** | A 股、美股（需自行获取数据） |
| **调用方式** | Python API，配置化驱动 |
| **GitHub Stars** | 31K |

**⚠️ 关键分析：Qlib 的"秒级响应"是有前提条件的**

需求文档 v1.0 将 Qlib 描述为"快（秒级）"，这个描述成立的前提是：**信号已经预先计算好，插件只是做查询**。实际上，QlibPlugin 需要 SEA 额外管理以下生命周期：

1. **数据初始化**：需下载 CSI300/CSI500 历史数据集（数 GB），或对接自定义数据源
2. **模型训练**：需训练一个 ML 模型（LightGBM/LSTM 等）
3. **信号生成**：需对股票池跑一次全量预测，生成信号序列，才能支持后续的 ticker 查询
4. **信号过期管理**：信号随时间推移失效，需定期重新生成

**Alpha 信号 → Direction 映射问题**：Qlib 的原始输出是预测收益率（Alpha 信号），一个浮点数，不是买/卖/持有。映射为 Direction 枚举需要阈值策略（如 Alpha > x% → BUY），该阈值应暴露为插件配置项。

**正确的使用模式**：Qlib 插件对外暴露标准 `evaluate(ticker, date)` 接口，但内部维护两层操作语义：
- **离线层**：数据下载、模型训练、全量信号预测（周期性批量执行）
- **在线层**：从本地信号库查询指定 ticker 的 Alpha 信号，秒级响应

详见技术文档第 3 章"Qlib+RD-Agent 双层架构设计"。

### 3.7 Microsoft RD-Agent — 范式不匹配（需重新定位）

| 属性 | 说明 |
|---|---|
| **技术路线** | LLM 驱动的自动化 R&D 框架，Qlib 为后端执行引擎 |
| **评估方法** | 自动读取金融文献 → 假设生成 → 因子代码生成 → Qlib 回测验证 → 迭代优化 |
| **输出内容** | 新发现的 Alpha 因子、因子表现报告、策略建议 |
| **调用方式** | `rdagent fin_quant`（接受市场配置 + 时间范围，不接受 ticker） |
| **耗时** | 数十分钟到数小时，每次运行约 $10（LLM API 费用） |

**❌ 根本问题：输出粒度不匹配**

RD-Agent 的输出粒度是"因子"（作用于整个股票池的函数），而非"个股评估"。它无法回答"评估 600519.SS"，只能回答"发现能预测 A 股收益的新因子"。

强行纳入 `evaluate(ticker, date)` 接口的链路极其间接：
```
触发 RD-Agent 完整研究运行（数十分钟~小时）
  → 等待"发现"若干因子
  → 用这些因子在 Qlib 里跑预测
  → 从预测结果里提取 ticker 对应的信号
  → 最后转成 Direction
```
这个链路有四层，输出的评估本质上来自 Qlib，不是 RD-Agent 本身。

**✅ 重新定位：从"实时评估插件"调整为"周期性因子研究任务"**

RD-Agent 更适合作为一个**周期性后台任务**（如每周运行一次，发现新因子并更新 Qlib 的模型），而非 Dispatcher 分发链上的实时评估节点。

SEA 中的正确角色设计：
- **RDAgentResearchJob**（离线）：定期触发，产出因子 → 注入 Qlib 模型 → 更新信号库
- **QlibQuantPlugin**（在线）：基于最新模型和信号库，响应 `evaluate(ticker, date)` 查询

RD-Agent 不直接面向 `evaluate()` 调用，而是作为 Qlib 插件的"能力增强器"在后台运行。

### 3.8 Qlib 与 RD-Agent 的关系

```
┌─────────────────────────────────────────────────────┐
│                  RD-Agent（自动化研究层）              │
│  读文献 → 提假设 → 生成因子代码 → 触发 Qlib 回测      │
│  强依赖 Qlib 环境 · 无 Qlib 无法运行                  │
└──────────────────────┬──────────────────────────────┘
                       │ 调用
┌──────────────────────▼──────────────────────────────┐
│                  Qlib（基础设施层）                    │
│  数据处理 · 特征工程 · ML 模型训练 · 回测引擎 · 信号存储 │
│  31K stars · 独立可用 · 无 RD-Agent 也能运行           │
└─────────────────────────────────────────────────────┘
```

- **RD-Agent 是 Qlib 的"上层自动化大脑"**，将量化研究过程自动化
- **Qlib 是 RD-Agent 的"底层执行引擎"**，提供数据、训练、回测能力
- RD-Agent(Q) 将研究过程拆分为 Research 阶段（提出假设和任务规划）和 Development 阶段（Co-STEER 代码生成 Agent），然后在 Qlib 的真实市场回测环境中执行
- **没有 Qlib，RD-Agent 根本无法运行；反之 Qlib 完全可以独立使用**

### 4.2 架构关键变化（相比 v1.0）

1. **插件从进程内调用改为 HTTP 服务调用**：每个插件独立 Docker 容器，通过 HTTP 接口与 SEA Core 通信
2. **Qlib 和 RD-Agent 合并为同一容器**：两者共享 Python 环境和数据目录，统一对外暴露接口
3. **区分"在线评估插件"与"离线研究任务"**：RD-Agent 不再作为 Dispatcher 分发链上的实时评估节点
4. **新增候选插件**：FinRobot、FinGPT-Forecaster 作为未来扩展选项

### 4.3 核心模块说明

#### EvaluationTask（评估任务）

用户发起的一次评估请求的数据结构，包含：
- 标的代码列表（如 `["600519.SS", "AAPL"]`）
- 评估日期
- 启用的插件列表（`null` 表示使用所有已启用插件）
- 各插件的可选配置覆盖

#### PluginRegistry（插件注册表）

负责插件的生命周期管理：
- 插件的注册（install）、启用（enable）、禁用（disable）、卸载（uninstall）
- 插件元信息的持久化（存储于本地配置文件 `plugins.yaml`）
- 插件可用性检查（**修订**：不仅是检查依赖是否安装，还需检查实际运行条件是否满足，如 Qlib 插件需检查信号库最后更新时间）

#### Dispatcher（调度器）

负责将 EvaluationTask 分发给已启用的插件：
- 通过 HTTP 调用各插件服务的 `/evaluate` 接口
- 支持**并发执行**（`asyncio.gather` + `httpx.AsyncClient`）
- 支持**超时控制**（单插件超时不影响整体结果）
- 支持**降级策略**（插件失败时记录错误，不中断整体流程）

#### Aggregator（结果聚合器）

将多个插件的 `PluginResult` 合并为 `EvalReport`：
- 标准化各插件的方向建议（买/卖/持有 → `+1/0/-1`）
- 计算加权综合评分（权重可配置）
- 标识各插件结论的共识度与分歧度
- 不做最终投资决策，仅呈现综合视图

## 7. 插件接口规范

### 7.1 核心接口定义

每个评估源插件必须实现以下 Python 抽象基类：

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional
from datetime import date
from enum import Enum

class Direction(Enum):
    BUY   = 1    # 看多
    HOLD  = 0    # 持有/中性
    SELL  = -1   # 看空

@dataclass
class PluginResult:
    plugin_id: str              # 插件唯一标识
    ticker: str                 # 标的代码
    eval_date: date             # 评估日期
    direction: Direction        # 方向建议
    confidence: float           # 置信度 [0.0, 1.0]
    summary: str                # 简要结论（≤ 200字）
    detail: Optional[dict]      # 原始详细输出（各插件自定义结构）
    latency_ms: int             # 执行耗时（毫秒）
    error: Optional[str] = None # 执行错误（None 表示成功）
    # 新增字段
    signal_age_days: Optional[int] = None  # 信号生成距今天数（Qlib 等预计算插件适用）

class BaseEvaluatorPlugin(ABC):

    @property
    @abstractmethod
    def plugin_id(self) -> str:
        """插件唯一标识，如 'tradingagents', 'qlib'"""

    @property
    @abstractmethod
    def display_name(self) -> str:
        """展示名称，如 'TradingAgents (TauricResearch)'"""

    @property
    @abstractmethod
    def supported_markets(self) -> list[str]:
        """支持的市场，如 ['US', 'A_SHARE', 'ALL']"""

    @abstractmethod
    def is_available(self) -> bool:
        """检查插件依赖是否满足（用于健康检查）"""

    @abstractmethod
    async def evaluate(self, ticker: str, eval_date: date, **kwargs) -> PluginResult:
        """执行评估，返回标准化结果"""

    def get_default_config(self) -> dict:
        """返回插件默认配置（可选实现）"""
        return {}
```

## 8. 结果聚合规范

### 8.1 EvalReport 结构

```python
@dataclass
class EvalReport:
    ticker: str
    eval_date: date
    generated_at: datetime

    # 各插件原始结果
    plugin_results: list[PluginResult]

    # 聚合评分（仅当 ≥2 个插件成功返回时计算）
    aggregated_score: Optional[float]   # [-1.0, 1.0]，负值看空，正值看多
    consensus_level: Optional[str]      # "strong" / "moderate" / "diverged"
    direction_summary: Optional[str]    # "2买/1持/0卖" 等文字描述

    # 执行统计
    total_plugins: int
    succeeded_plugins: int
    failed_plugins: int
    total_latency_ms: int

    # 新增：信号质量标注
    signal_quality_notes: Optional[list[str]]  # 如 ["Qlib 信号已过期 15 天"]
```

### 8.2 聚合算法

综合评分 = Σ（插件方向值 × 置信度 × 权重） / Σ（权重）

其中：
- 插件方向值：BUY=+1, HOLD=0, SELL=-1
- 权重在 `config.yaml` 中配置（默认各插件权重相等）
- 失败插件不参与聚合计算
- **修订**：信号过期插件（如 Qlib STALE 状态）可配置为降权参与或排除

**共识度判定：**

| 条件 | 共识度 |
|---|---|
| 所有成功插件方向一致 | `strong`（强共识） |
| >60% 插件方向一致 | `moderate`（中等共识） |
| 无多数方向 | `diverged`（意见分歧） |

### 8.3 Qlib Alpha 信号 → Direction 映射规则（新增）

Qlib 插件输出的原始 Alpha 信号（预测收益率）需通过阈值策略映射为 Direction：

| Alpha 信号范围 | Direction | 说明 |
|---|---|---|
| Alpha > `alpha_buy_threshold`（默认 +2%） | BUY | 预测收益显著为正 |
| `alpha_sell_threshold` ≤ Alpha ≤ `alpha_buy_threshold` | HOLD | 预测收益在中间区域 |
| Alpha < `alpha_sell_threshold`（默认 -2%） | SELL | 预测收益显著为负 |

阈值在插件配置中可调，置信度基于 Alpha 信号在股票池中的排名百分位计算。

### 9.1 总体部署方案

```
宿主机（Mac Mini / Linux Server）
│
├── Docker Compose: sea_network
│   │
│   ├── sea-core              (FastAPI, port 8080)

│   │   └── POST /evaluate
│   │
│   ├── qlib-rdagent-svc      (FastAPI, port 8083)
│   │   ├── POST /evaluate         （在线信号查询）
│   │   ├── POST /research/trigger （离线研究触发）
│   │   └── GET  /research/status  （研究状态查询）
│   │   └── 挂载：/var/run/docker.sock（DooD）
│   │
│   └── (future) finrobot-svc (FastAPI, port 8084)
```

### 9.2 各插件部署要求

| 插件服务 | 基础镜像 | 特殊要求 | 资源需求 |
|---|---|---|---|
| sea-core | `python:3.11-slim` | 无 | 最小 |
| tradingagents-svc | `python:3.11-slim` | 需安装 tradingagents 包 | 内存 ~2GB |
| tradingagents-cn-svc | `python:3.11-slim` | 需安装 tradingagents-cn 包 | 内存 ~2GB |
| qlib-rdagent-svc | `pyqlib/qlib_image_stable:stable` | 叠加安装 rdagent；**必须挂载 Docker socket（DooD）**；仅 Linux 环境完整可用 | 内存 ~8-12GB（训练时），闲时 ~2GB |
| finrobot-svc（候选） | `python:3.11-slim` | 需安装 finrobot 包 | 内存 ~2GB |

### 9.3 关键部署约束

1. **DooD（Docker-outside-of-Docker）模式**：qlib-rdagent-svc 容器需挂载宿主机 Docker socket，使 RD-Agent 能在宿主机上动态创建 `local_qlib` 兄弟容器来隔离执行因子代码。**严禁使用 `--privileged` 的 DinD 模式**。
2. **卷路径一致性**：DooD 模式下，bind mount 的源路径必须从宿主机 Docker Daemon 的视角可解析。qlib-rdagent-svc 容器内的工作目录需通过 Named Volume 或 bind mount 映射到宿主机固定路径。
3. **Mac 上额外注意**：Docker Desktop on Mac 的 socket 路径建议使用 `/var/run/docker.sock.raw`；RD-Agent 在 Mac 上非官方支持，可能遇到兼容性问题，建议开发阶段先在 Mac 上验证 SEA Core + LLM 类插件，qlib-rdagent-svc 等上 Linux 或云服务器后再接入。

### 9.4 硬件要求

| 组件 | 最低要求 | 推荐配置 |
|---|---|---|
| SEA Core + LLM 插件 | 4GB 内存，10GB 磁盘 | 8GB 内存 |
| Qlib 查询层 | 4GB 内存，20GB 磁盘 | 8GB 内存 |
| Qlib 训练（LightGBM，CSI300） | 8-12GB 内存，20GB 磁盘 | 16GB 内存 |
| RD-Agent 研究任务 | Linux 系统 + Docker，8GB 内存 | 16GB 内存 + 云服务器 |

**Mac Mini（16GB 内存 / 256GB 磁盘）可行性**：
- ✅ SEA Core + LLM 类插件：完全可行
- ✅ Qlib 查询层 + LightGBM 训练（CSI300 日线）：可行，内存峰值约 8-12GB
- ⚠️ RD-Agent：非官方支持 macOS，建议通过方案二（云上运行）或方案一（Docker Desktop on Mac 尝试）解决
- ❌ 分钟线数据：不要下载，磁盘不够（需 80-150GB）

## 12. 风险与约束

### 12.1 依赖风险

| 风险 | 描述 | 缓解措施 |
|---|---|---|
| 上游项目 API 变更 | TradingAgents 等项目仍在快速迭代，接口可能变化 | 插件层做版本适配；在 `plugin.yaml` 中声明兼容版本范围 |
| Qlib 数据依赖 | Qlib 需要本地预先下载数据集，冷启动成本高 | 提供数据初始化脚本；文档中明确说明；Phase 3 专项处理 |
| RD-Agent 平台限制 | 仅官方支持 Linux，Mac 用户有障碍 | 文档明确说明；提供云上运行方案；Phase 3 优先解决 Linux 部署 |
| RD-Agent 执行时间 | 单次运行可能耗时数十分钟到数小时 | 设计为异步离线任务，不阻塞在线评估；超时后降级 |
| LLM API 费用 | RD-Agent 每次运行约 $10；TradingAgents 每次也有 API 成本 | 文档中透明说明成本；支持本地模型（Ollama）降本 |

### 12.2 设计约束

- **SEA 不持有任何评估逻辑**：所有评估判断由插件（开源项目）完成，SEA 只做转发和展示
- **SEA 不提供交易执行能力**：仅输出评估结论，无任何下单或资金操作
- **免责声明**：SEA 聚合的所有评估结论仅供研究参考，不构成投资建议

### 12.3 Docker 特定风险（新增）

| 风险 | 描述 | 缓解措施 |
|---|---|---|
| DooD 卷路径问题 | RD-Agent 容器内路径与宿主机路径不一致导致 bind mount 失败 | 使用 Named Volume 或环境变量传递宿主机路径；技术文档中详细说明 |
| Mac Docker Desktop 兼容性 | Docker Desktop on Mac 的 socket 路径和网络模式与 Linux 有差异 | 开发阶段 Mac 仅运行 LLM 类插件；qlib-rdagent-svc 在 Linux 环境验证 |
| 容器资源竞争 | 多容器同时运行可能超出 Mac Mini 16GB 内存 | Docker Compose 中配置内存限制；文档给出各插件资源需求 |

## 13. 已决策事项

以下事项在 v1.0 中标记为"开放问题"，经分析讨论后已作出决策：

### 13.1 A 股 vs 美股路由策略

**决策**：在 Dispatcher 层实现标的-插件路由逻辑：
- A 股标的（`.SS` / `.SZ` 后缀）：TradingAgents-CN 和 TradingAgents 同时启用时，优先使用 CN 版；仅其中之一启用时使用启用的那个
- 海外标的：仅使用 TradingAgents 原版
- 用户可通过配置覆盖此行为（如强制对 A 股同时跑两个以获取交叉验证）

### 13.2 RD-Agent 定位

**决策**：RD-Agent **不**作为 Dispatcher 分发链上的实时评估插件。它被重新定位为：
- **离线角色**：`RDAgentResearchJob` — 周期性后台任务（每周/每月），产出因子 → 更新 Qlib 模型 → 更新信号库
- **在线角色**：`QlibQuantPlugin` — 基于最新信号库响应 `evaluate(ticker, date)` 查询
- 在需求文档和架构图中明确区分"在线评估插件"和"离线研究任务"两种角色

### 13.3 Qlib 运行模式

**决策**：采用"预计算信号 + 实时查询"模式：
- 离线批量生成全量信号（周期性或事件触发）
- 在线查询时直接从本地 SignalRecord 读取，秒级响应
- 不采用每次 evaluate 时实时跑模型的方式（耗时长且不可预测）

### 13.4 插件部署模式

**决策**：采用 Docker 容器 + HTTP 通信模式：
- 每个插件独立 Docker 容器，暴露 HTTP API
- SEA Core 通过 `httpx.AsyncClient` 并发调用
- 本地开发阶段可保留进程内直接调用模式（通过配置切换）
- Qlib 和 RD-Agent 放入同一容器（共享 Python 环境和数据目录）

### 13.5 结果缓存策略

**决策**：
- Qlib 信号查询结果：可缓存（同一天同一标的信号相同），缓存 TTL = 24h
- LLM 类插件结果：不缓存（每次运行结论可能不同）
